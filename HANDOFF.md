## PIPELINE
Single command from repo root:
```bash
./render_pipeline.sh rgbgrid-medium
```

Steps:
1. Reads rgbgrid-medium/config.json
2. Validates project name matches config.project.name
3. Runs build_scene.py (if pipeline.build_scene.enabled=true)
4. Runs generate.py (if any tree in scene.trees[] has enabled=true)
5. Runs pbrt with --gpu --stats flags
6. Archives bundle to local Archive/
7. Syncs to Google Drive (if pipeline.rclone_sync.enabled=true)

## FILE ROLES
- generate.py — orchestrator, called by render_pipeline.sh
  - imports space_col and foliage
  - iterates over scene.trees[] list
  - for each enabled tree: runs Tree3D, writes tree_N.pbrt
  - for each enabled tree with foliage: runs foliage.run(), writes foliage_N.pbrt
  - no serialization between modules
- space_col.py — space colonization tree algorithm
  - generates tree_N.pbrt (cylinders + joint spheres)
  - write_tree() takes index=N parameter for indexed output filename
  - _inject_attraction_points() — NEW: continuously added attraction points
    during growth, controlled by continuous_attraction config block
- foliage.py — foliage generation
  - run() and write_foliage() take index=N parameter
  - ObjectBegin names are "tree_N_leaf_M" — globally unique across all foliage files

## KEY pbrt-v4 LESSONS LEARNED
- Include for rgbgrid medium goes INSIDE WorldBegin
- MediumInterface must come BEFORE Material in AttributeBegin block
- Box container uses trianglemesh with inward-facing winding
- Cylinder shape has no end caps — use spheres at joints
- rgbgrid supports volumetric emission via "rgb Le" and "float Lescale"
- Instancing: ObjectBegin/ObjectEnd defines geometry once, ObjectInstance places it
- Materials are BAKED into ObjectBegin — cannot vary per instance
- For per-leaf color variation: define N ObjectBegin variants, randomly assign instances
- foliage_N.pbrt can exceed 100MB — keep in .gitignore
- tree_N.pbrt also gitignored
- Distant light correct for sunlight (parallel rays, from/to)
- Grid enabled flag skips both rgbgrid.pbrt generation and Include in scene
- pbrt-v4 does NOT support nested ObjectBegin — confirmed blocker for forest instancing
- ObjectBegin names must be globally unique across ALL included files

## COMPETING-FOREST BRANCH — CURRENT STATE

### Active work: Runions algorithm improvements
All parameter exploration and algorithm improvements are on competing-forest branch.
Goal: emulate Runions Figure 8 (mature tree with branch hierarchy) and
Figure 12 (close-up with massive trunk, bark texture, leaves on fine twigs).

### Key insight from Runions papers:
- D = 1 is Runions' normalized unit. All parameters expressed as multiples of D.
- dk_multiplier is the coefficient in dk = dk_multiplier × D
- di_multiplier is the coefficient in di = di_multiplier × D
- birth_dist is the minimum spacing between attraction points

### Current working config (competing-forest, single tree, no foliage):
```json
"D": 1.0,
"dk_multiplier": 2.0,
"di_multiplier": 8.0,
"birth_dist": 4.0,
"num_leaves": 500,
"point_cloud_center": [0.0, 26.0, 0.0],
"point_cloud_radius": 20.0,
"base_radius": 0.015,
"joint_radius_multiplier": 1.3,
"joint_radius_cap": 0.1,
"root_position": [0.0, 0.0, 0.0],
"continuous_attraction": {
    "enabled": true,
    "points_per_iteration": 200,
    "initial_birth_dist": 4.0,
    "final_birth_dist": 0.3,
    "iterations_to_full_density": 100
}
```

### Camera (competing-forest):
```json
"eye": [0, 20.0, 80.0],
"look": [0, 20.0, 0]
```

### Continuous attraction points — IMPLEMENTED (WIP):
- _inject_attraction_points() added to Tree3D in space_col.py
- Called each iteration inside grow() when continuous_attraction.enabled=true
- Linearly interpolates birth_dist from initial_birth_dist to final_birth_dist
- Produces branch hierarchy: large limbs first, fine twigs later
- Current result: promising — clear trunk, major limbs, fine twig detail at crown
- NOT yet matching Runions Figure 8 quality

### What Figure 8 and 12 have that we don't yet:
1. Massive trunk taper — trunk enormous at base, Murray's law needs tuning
2. Fewer, coarser primary limbs — too many thin branches from start
3. Much longer visible trunk before first branching
4. Decurrent form — trunk splits into major limbs low down
5. Bark texture on trunk and branches (material/rendering work, deferred)
6. Leaves only on fine outer twigs, inner branches bare

### Immediate next steps (competing-forest):
1. Tune trunk thickness — increase base_radius, tune Murray's law parameters
2. Reduce number of primary limbs — increase initial_birth_dist further
3. Extend trunk length — adjust point_cloud_center Y further from root
4. Surface bias for attraction points — concentrate points near crown envelope
   (Runions Figure 7) — not yet implemented
5. Bark texture — deferred until geometry is correct

### Forest system — Phase 2 (planned, after tree quality is good):
- Competing trees sharing one attraction point cloud (Runions Figure 10)
- Each tree tagged by tree_index on Branch3D
- After growth, partition branches by tree_index
- TreeProxy object adapts partition to foliage.run() interface
- When forest instancing active, foliage written as flat geometry (no nested ObjectBegin)
- forest.enabled=false falls back to current multi-tree pipeline exactly
